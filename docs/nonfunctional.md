# Дополнительная спецификация
## Командный проект «Цветочная оранжерея» (вариант 8)

### Нефункциональные требования

#### 1. Надёжность
- **NR1.1** Система должна корректно обрабатывать ошибки SQL и выводить понятные сообщения.
- **NR1.2** При нарушении ограничений целостности (например, удаление растения, входящего в композицию) система должна предотвращать операцию или выполнять каскадное удаление.
- **NR1.3** Система должна корректно завершать работу при любых ошибках (без утечек памяти, закрывая соединение с БД).

#### 2. Производительность
- **NR2.1** Время отклика на пользовательский ввод не должно превышать 2 секунд.
- **NR2.2** Запросы к базе данных должны выполняться за < 0.5 секунд для таблиц до 10 000 записей.

#### 3. Удобство использования
- **NR3.1** Интерфейс должен быть консольным с понятными меню и подсказками.
- **NR3.2** Пункты меню должны быть пронумерованы для быстрого выбора.
- **NR3.3** При ошибках система должна выводить понятные сообщения на русском языке.
- **NR3.4** При вводе данных должны быть подсказки о формате (например, даты в формате YYYY-MM-DD).

#### 4. Сопровождаемость
- **NR4.1** Код должен быть организован по модели КИС (Клиент-Интерфейс-Сервер).
- **NR4.2** Каждый модуль должен быть в отдельном файле с соответствующим заголовочным файлом.
- **NR4.3** Код должен быть документирован (комментарии к функциям и сложным участкам).
- **NR4.4** Проект должен собираться с помощью Makefile.

#### 5. Безопасность
- **NR5.1** Пароли пользователей должны храниться в хешированном виде (SHA-256).
- **NR5.2** Доступ к функциям должен разграничиваться по ролям.
- **NR5.3** Администратор не может быть удалён.
- **NR5.4** Поля ввода должны защищать от SQL-инъекций (использование параметризованных запросов).

#### 6. Портативность
- **NR6.1** Система должна работать на macOS, Linux и Windows (с WSL).
- **NR6.2** Все пути должны быть относительными для возможности переноса.

#### 7. Тестирование
- **NR7.1** Проект должен иметь unit-тесты (минимум 3 теста на каждый модуль).
- **NR7.2** Тесты должны запускаться автоматически через GitHub Actions.
- **NR7.3** Код должен проходить все тесты без ошибок.

### Функциональные требования, общие для нескольких вариантов использования

| Требование | Описание | Варианты использования |
|------------|----------|------------------------|
| **FR-COMMON-1** | Все операции CRUD должны подтверждаться сообщением об успехе или ошибке | Добавление растения, удаление композиции, редактирование заказа и т.д. |
| **FR-COMMON-2** | Поиск по шаблону должен поддерживать подстановку `%` | Поиск заказов по клиенту, поиск растений |
| **FR-COMMON-3** | Все даты должны храниться в формате YYYY-MM-DD | Заказы, дата выполнения |
| **FR-COMMON-4** | При удалении записи с внешними ключами должно выполняться каскадное удаление или выдаваться ошибка | Удаление растения → удаление из композиций, удаление композиции → удаление из заказов |
| **FR-COMMON-5** | Пользователь должен видеть свою роль в главном меню | Все варианты использования |
